## 1. 最新资料研究结论

本轮参考了一手或官方资料，并转化为项目规则：

- OpenAI Agents SDK 将多代理协作归纳为两类常见模式：集中式 manager / agents-as-tools，以及 handoffs。RepairDesk 默认采用 manager 模式：主线程保留集成权，子代理作为专业部门提供结果。
- OpenAI Handoffs 文档强调 handoff 适合让专门 agent 接管特定任务，但需要控制传入上下文、工具描述和 handoff 输入。RepairDesk 因代码集成风险较高，不让子代理自动接管最终写入。
- Claude Code subagents 文档强调：子代理适合隔离高噪声搜索、日志和文件扫描；要限制工具权限、写清 description、让专门 agent 只做专门任务。RepairDesk 因此规定子代理默认 read_only。
- Microsoft AutoGen Teams 文档建议：复杂任务才使用 team；简单任务先优化单 agent，不要过早引入团队。RepairDesk 因此保留“默认单代理、必要时多代理”的原则。
- AutoGen 的 team termination / critic 模式说明，多代理必须有明确停止条件和验收信号。RepairDesk 的停止条件是：主线程完成集成、验证通过、风险说明清楚。

资料入口：

- OpenAI Agents SDK: https://openai.github.io/openai-agents-python/agents/
- OpenAI Handoffs: https://openai.github.io/openai-agents-python/handoffs/
- Claude Code Subagents: https://code.claude.com/docs/en/sub-agents
- Microsoft AutoGen AgentChat / Teams: https://microsoft.github.io/autogen/stable/user-guide/agentchat-user-guide/tutorial/teams.html

## 2. 核心运行规则

每个非简单任务先做部门化 agenda pass：

1. 明确用户目标。
2. 明确限制：只读、不要改文件、不要运行命令、不要联网、不要改数据库等。
3. 判断是否需要联网研究。
4. 分类业务域：orders、customers、inventory、messages、settings、platform、buyback、shared、server、app routes。
5. 分类技术域：产品流程、数据、UI、前端、API、测试、安全、文档、集成。
6. 评估风险等级：low、medium、high。
7. 决定单代理还是多代理。
8. 给每个子代理分配部门、权限模式和交付物。
9. 主线程集成所有结论。
10. 运行必要验收。

默认行为：

- 用户只把任务交给 Integration Lead。Integration Lead 是唯一面向用户的决策人，负责重新分析任务、分配部门、组织争论、裁决冲突和最终汇报。
- 小型、局部、低风险任务使用单代理。
- 用户明确要求多代理、复核、部门管理、模拟验收时，启用多代理。
- 子代理默认只读。
- 子代理只有在主线程给出明确文件范围时才能 scoped_write。
- 主线程永远是最终集成负责人。
- 子代理默认并发上限为 3，建议保持 2-4 个活跃代理；硬上限 5。
- 并行写入默认只允许 1 个写入代理，硬上限 2；两个写入代理不得拥有同一文件或目录范围。
- 完成的子代理必须及时关闭，释放后才能继续生成新的子代理。
- 子代理不得直接向用户请求扩大权限或改变范围；缺上下文、缺权限、发现冲突时，统一回报给 Integration Lead。

## 3. 联网研究规则

### 必须联网的情况

遇到以下任一情况，主线程必须联网搜索最新资料：

- 用户明确要求“联网”“搜索最新资料”“查最新方案”。
- 涉及 AI agent、自动化、框架、SDK、平台能力、云服务、Supabase、Vercel、OpenAI、Claude、Google、Microsoft 等可能变化的技术。
- 涉及法律、合规、安全、隐私、支付、通信平台政策、WhatsApp API、打印设备兼容性等高变化领域。
- 涉及推荐采购、设备选型、外部服务选型、价格或可用性。
- 当前仓库规则和外部平台文档可能冲突。

### 优先资料来源

按优先级使用：

1. 官方文档、官方博客、官方 GitHub。
2. 标准、规范、RFC、厂商 API reference。
3. 学术论文或技术报告，用于补充设计趋势，不直接替代官方 API。
4. 高质量工程文章，用于启发，不作为最终规则来源。

### 联网研究输出

联网研究完成后，必须在计划或最终答复中说明：

- 查了哪些来源。
- 对本项目产生什么规则变化。
- 哪些只是外部启发，哪些已经写入本项目规则。
- 如果资料和本地代码冲突，以本地代码和项目规则为准，除非用户明确要求升级。

风险等级：

- medium：单一 feature、表单、查询缓存、响应式布局、API facade。
- high：订单状态机、付款、客户 PII、权限、库存变动、数据库结构、跨模块重构。

## 5. 是否启用多代理

满足任一条件时启用多代理：

- 用户明确要求子代理、多 IE agent、多代理、部门执行、复核或模拟验收。
- 任务跨两个以上业务域。
- 任务同时涉及 UI、数据/API、测试、架构边界中的两个以上方面。
- 任务改变订单工作流、付款、权限、客户数据、库存或数据库结构。
- 任务需要并行审计多个页面、失败测试、流程或回归风险。
- 任务需要模拟前台、技师、店长、客户等不同角色。
- 任务需要联网研究并转化为项目长期规则。

保持单代理的情况：

- 局部且明确的小修复。
- 单文件样式或文案。
- 用户要求快速直接处理。
- 多代理会重复读取同一文件，增加协调成本。

## 6. 部门体系

部门是 AI 工作分配角色，不是公司真实组织架构。

### 6.1 集成负责人

标签：INT

职责：

- 任务入口分诊。
- 部门分配。
- 子代理 prompt 和权限控制。
- 冲突解决。
- 最终代码集成。
- 最终验证和用户报告。

触发：

- 每个任务都触发。
- 跨域任务必须显式执行。

验收：

- 每个部门有明确目标。
- 没有两个写入代理编辑同一范围。
- 最终结果符合用户最新请求。

### 6.2 产品流程部

标签：FLOW、SCOPE

职责：

- 用户故事和业务流程。
- 工单、客户、库存、报价、付款、取机、售后状态规则。
- 技师、前台、店长、客户视角任务路径。
- 前置条件、下一步动作和异常路径。

触发：

- 任何状态、流程、权限、归属字段、扫码任务、打印任务变更。

交付物：

- 流程表。
- 允许动作和禁止动作。
- 状态迁移预期。
- 业务验收标准。

### 6.3 数据设计部

标签：DATA、DB

职责：

- 数据模型、字段、枚举、关系。
- Supabase migration。
- mock fixtures 和 test builders。
- `src/lib/repairdesk/types.ts` 类型契约。
- 查询性能、索引、分页、统计口径。

触发：

- 新字段、新表、新枚举、新指标、API payload 变化、mock/server 不一致。

验收：

- 数据库、server、client、mock、测试一致。
- client component 不导入 server-only 模块。
- loading、empty、error 状态不破坏。

### 6.4 页面设计部

标签：UX、UI

职责：

- 信息架构。
- 响应式布局。
- Dialog、Sheet、table、card、list、dense workspace。
- 移动端技师任务体验。
- 视觉一致性、动画、交互状态。

必须读取：

- `docs/UI_PAGE_GENERATION_DECLARATION.md`
- `docs/COMPONENT_GENERATION_DECLARATION.md`
- `docs/RESPONSIVE_DENSITY_PLAN.md`
- `src/lib/ui-patterns.ts`
- `src/lib/component-patterns.ts`
- `src/styles.css`

验收：

- 无页面级横向溢出。
- 移动端保持高密度、任务导向。
- 文本和控件不重叠。
- 颜色来自 design tokens。

### 6.5 前端实现部

标签：FE

职责：

- Next.js App Router route body。
- feature screens 和 client components。
- React Query 与 invalidate。
- 表单状态、Dialog、Sheet、列表、局部 UI 状态。

验收：

- `src/app/*` 保持 thin。
- client 取数走 `@/lib/repairdesk/api` 或 feature API facade。
- query key 优先使用 `ordersKeys`、`customersKeys` 等工厂。
- UI primitives 来自 `src/components/ui/*`。

### 6.6 后端/API 部

标签：API、SERVER

职责：

- Route handlers。
- server repository/service。
- zod schemas。
- API facade。
- 服务端业务规则校验。

验收：

- 业务规则不只靠前端限制。
- server-only 代码不进入 client。
- 错误结构可被 UI 展示。
- mutation 后能正确刷新相关 query。

### 6.7 QA 验收部

标签：QA、TEST

职责：

- 测试计划。
- 静态门禁。
- 浏览器和移动端验证。
- 角色模拟。
- 回归风险说明。

默认门禁：

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

如果项目提供并覆盖全部门禁，可使用：

```bash
npm run check
```

验收：

- 说明运行了什么。
- 说明没运行什么以及原因。
- UI 改动必须尽量做移动端和桌面关键路径检查。

### 6.8 安全隐私部

标签：SECURITY、PRIVACY

职责：

- auth、权限、客户资料、付款、消息、附件、环境变量。
- secret handling。
- 最小权限原则。

触发：

- 涉及客户 PII、员工身份、付款、Supabase 凭据、WhatsApp 内容、文件上传。

验收：

- 不提交 secret。
- PII 不被不必要地放进日志、公开路由、二维码 payload 或 browser storage。
- 高权限动作有服务端校验。

### 6.9 文档规则部

标签：DOC、RULES

职责：

- 项目声明。
- 架构规则。
- AI 部门管理规则。
- release 和迁移说明。

触发：

- 引入可复用模式。
- 规则改变或冲突。
- 用户要求“之后都按这个来”。

验收：

- 规则必须可执行。
- 不写空泛偏好。
- 不与 `AGENTS.md` 或 `docs/ARCHITECTURE.md` 冲突。

## 7. 子代理任务包

每次分配子代理必须提供：

```txt
department:
mode: read_only | scoped_write
goal:
context:
must_read:
allowed_files:
forbidden_files:
expected_output:
acceptance:
deadline:
```

规则：

- `read_only` 是默认权限。
- `scoped_write` 必须写明允许修改的文件或目录。
- 子代理不得拥有最终 integration 权。
- 不得让两个写入代理改同一文件范围。
- 子代理发现超范围问题时，只能报告，不能自行扩大 scope。
- 子代理不得再创建子代理。
- 子代理不得执行最终 merge、stage、commit、push、deploy、destructive SQL、secret 读取或 secret 写入。

## 8. 权限模式

### read_only

允许：

- 读文件。
- 搜索代码。
- 检查结构。
- 输出 findings、风险、patch plan。

禁止：

- 编辑文件。
- 格式化文件。
- 安装依赖。
- 运行破坏性命令。

适用：

- 审计、规划、高风险区域、QA、安全、架构、需求不清。

### scoped_write

允许：

- 只修改明确指定的文件或目录。
- 运行非破坏性验证。

禁止：

- 自行扩大范围。
- 回滚无关改动。
- 触碰其他 worker 负责的文件。

适用：

- 边界清晰、文件范围不重叠的实现切片。

### integration_write

仅主线程使用。

职责：

- 应用最终改动。
- 合并或拒绝子代理建议。
- 解决冲突。
- 跑最终验证。
- 写最终用户总结。

## 9. 冲突优先级

出现规则或部门意见冲突时，按以下顺序：

1. 用户最新明确指令。
2. `AGENTS.md`。
3. 本文件。
4. `docs/ARCHITECTURE.md`。
5. UI、组件、响应式声明文档。
6. 现有代码模式。
7. 最小安全改动。
8. 单个子代理偏好。

如仍无法安全决定，主线程给出推荐方案；只有在合理假设风险很高时才向用户追问。

## 11. 实现规则

所有部门必须遵守：

- 使用 Next.js App Router：`src/app`。
- `src/app/*` 保持 thin。
- 交互页面主体放在 feature screens。
- order/customer 业务 UI 放在 `src/features/*`。
- shared pure helpers 放在 `src/shared/lib`。
- 跨 feature entity rules 放在 `src/entities/*`。
- client component 不得 import `src/server/*`。
- app data 使用 `@/lib/repairdesk/api` 或 feature API facade。
- React Query cache 优先使用 feature query key factory。
- UI 控件复用 `src/components/ui/*`。
- 金额、状态、电话优先复用现有业务渲染组件。
- design tokens 只来自 `src/styles.css`。
- 不重新引入 TanStack Router/Start 或 Vite entrypoints。

## 13. 持续改进记录

### Customer Search Verification Record

当任务触及客户搜索、客户查找弹层、客户历史工单入口、订单详情里的客户信息编辑或从客户上下文打开工单详情时，相关部门必须保留以下验收记录：

- DATA/API：客户搜索必须使用服务端过滤、轻量字段、明确 limit 和店铺隔离；禁止为了弹层搜索拉取完整客户列表或完整 PII payload 后在前端过滤。
- FE/UX：客户搜索弹层必须支持 idle、输入提示、loading、结果、empty、error、清空、Escape/外部关闭、键盘上下选择、Enter 选择、鼠标/触摸选择。
- QA：必须检查 `390x844`、`430x932`、`768x1024`、`1024x768`、`1280x800`、`1440x900` 中关键入口无页面级横向溢出，至少确认 `document.documentElement.scrollWidth <= window.innerWidth`。
- Order Detail：从订单详情编辑客户姓名/主电话时，归属字段、技师、录入人仍保持只读；当前 API 未支持真正重绑 `customer_id` 时，搜索选择只能带入姓名和电话，不得暗中改变订单归属。
- Cache：客户搜索使用 `customersKeys.search(query, limit)`，客户设备读取使用 `customersKeys.devices(customerId)`；客户创建、更新、工单创建或编辑后需要刷新相关 customers / orders query。
- Regression：金额编辑仍必须保持 string draft 行为；空金额不能被强制变成 `0`，保存后的总额、尾款、payload 必须一致。
- Record：如果验收发现新问题，必须在本节追加日期、问题、影响范围和以后规则变化。
